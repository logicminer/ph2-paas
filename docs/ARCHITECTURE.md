# Architecture

## Overview

PH2 PaaS is a multi-tenant hosting platform that serves WordPress (via OpenLiteSpeed) and Node.js (via Dokku) applications from a single baremetal server, reachable through Cloudflare Tunnel without requiring a public IP.

## The four layers

### 1. Edge layer — Cloudflare Tunnel

```
Internet → Cloudflare edge (TLS terminates) → cloudflared (outbound tunnel) → localhost:80
```

- **cloudflared** runs as a systemd service, maintaining a persistent outbound connection to Cloudflare's edge.
- The tunnel has a **catch-all ingress**: every hostname with a CNAME pointing to the tunnel gets routed to `localhost:80`.
- Exception: the Portainer hostname gets a specific ingress rule to `https://localhost:9443` (with TLS verification disabled for OLS's self-signed cert).
- **No inbound ports required** — this is why it works behind CGNAT.
- TLS terminates at Cloudflare's edge. Between cloudflared and localhost, traffic is plaintext HTTP.

### 2. Routing layer — Dokku Nginx (port 80)

Dokku installs Nginx, which listens on port 80 as the single front door. It routes by hostname:

| Hostname pattern | Routes to | Config source |
|---|---|---|
| `<app>.<global-domain>` | Dokku app container | `/home/dokku/<app>/nginx.conf` (Dokku-managed) |
| WP client domains | OLS container (`127.0.0.1:8088`) | `/etc/nginx/wp-domains/<domain>.conf` (spawn-wp.sh generated) |
| Panel domain | Dokku panel container | Dokku-managed |

The WP gateway is a one-line include: `/etc/nginx/conf.d/wp-proxy.conf` → `include /etc/nginx/wp-domains/*.conf;`. Each spawned WP site gets its own `.conf` file dropped in that directory.

### 3. Application layer — two stacks

**WordPress stack (shared):**
- **OpenLiteSpeed (OLS)** — one container, serves all WP vhosts. OLS uses LSAPI (not PHP-FPM) for lower overhead, and LSCache for server-level full-page caching.
- OLS runs PHP as `nobody:nogroup` (uid 65534). WP files must be owned by `nobody` for uploads to work.
- Each WP site is an OLS vhost with a `vhconf.conf` (per-site config: docRoot, scriptHandler, rewrite).
- Port mapping: host `127.0.0.1:8088` → container `80`.

**Node.js stack (per-app):**
- **Dokku** — Heroku-style PaaS. `git push` triggers buildpack detection → build → container → deploy.
- Each app is isolated in its own container with its own port.
- Zero-downtime deploys with healthchecks.

### 4. Data layer — shared MariaDB

- **MariaDB 11.4** — one container on the `wp-tier` Docker network.
- Each WP site gets a logical database (`wp_<domain_with_underscores>`) + dedicated user.
- Memory-capped at 1GB with `memswap_limit=1g` (no swap — protects InnoDB buffer pool).
- Dokku Node apps can use their own databases (via Dokku MariaDB plugin) or external services.

## Memory tier

```
RAM (physical) → zram (compressed, priority 100) → disk swap (priority 10)
```

- **zram**: `lz4` compression, sized to match RAM (capped at 16GB). Used first when RAM is pressured. Effectively expands usable memory ~40% for compressible workloads (PHP/Node compress 2-3x).
- **Disk swap**: overflow-only (priority 10, `vm.swappiness=10`). Only used when zram is overrun.
- Config: `/etc/default/zramswap` + `/etc/fstab` swap entry + `/etc/sysctl.conf` swappiness.

## Management surfaces

| URL | What | Auth |
|---|---|---|
| `panel.<domain>` | Control panel (Next.js) — spawn/destroy/sites/dashboard | Cloudflare Access (email OTP) |
| `ph2.<domain>` | Portainer CE — container logs, exec, resource stats | Cloudflare Access (email OTP) |
| WP client sites | WordPress admin at `/wp-admin` | WordPress login |
| OLS WebAdmin | `localhost:7080` (not exposed) | OLS admin password |

## Credential flow

```
/etc/ph2/env (mode 600, root-only)
  ├── CLOUDFLARE_ACCOUNT_ID, CLOUDFLARE_TOKEN → used by spawn/destroy/backup scripts
  ├── MARIADB_ROOT_PASSWORD → used by scripts + stored in /opt/ols/.env for OLS
  ├── OLS_ADMIN_PASSWORD → stored in /opt/ols/.env
  ├── TUNNEL_ID → used by scripts + /etc/cloudflared/config.yml
  └── All paths, container names, version pins

/etc/cloudflared/credentials.json (mode 600, root-only)
  └── Tunnel cert/key (generated by tunnel creation API)

Panel SQLite (/var/lib/dokku/data/panel-data/panel.db)
  └── Per-site WP admin passwords + DB passwords (plaintext, backed up daily)
```

## Docker networking

```
wp-tier (external network)
  ├── mariadb (container, alias: mariadb)
  └── litespeed (container, member)

ols_default (bridge network)
  └── litespeed (also on this for OLS internal comms)

Dokku network (per-app)
  └── Each Dokku app gets its own network namespace
```

OLS reaches MariaDB via the hostname `mariadb` (Docker DNS on the `wp-tier` network).

## Backup architecture

Six components, daily at 02:00:

| Component | Format | Contains |
|---|---|---|
| databases | `.sql.gz` | All MariaDB databases (`--all-databases --single-transaction`) |
| wp-sites | `.tar.gz` | `/opt/ols/sites/` (uploads, themes, plugins, wp-config) — excludes cache |
| ols-config | `.tar.gz` | `/opt/ols/lsws/conf` (vhosts, listeners) |
| nginx-wp | `.tar.gz` | `/etc/nginx/wp-domains/` + `wp-proxy.conf` |
| panel | `.tar.gz` | Panel SQLite + Dokku app configs |
| cloudflare | `.json` | All zones + DNS records (API snapshot) |

Retention: 7 daily + 4 weekly (Sunday promoted, kept 28 days).

## Future: multi-node (Path A fleet)

The current architecture is single-node. Path A (planned) adds:

1. **Bootstrap script** (this repo) provisions new baremetals as worker nodes
2. **Controller upgrade** — panel learns about multiple nodes, picks one per spawn
3. **Remote execution** — controller SSHes to chosen worker
4. **Per-node tunnels** — each node gets its own tunnel; DNS points each site to its node's tunnel

Everything in this repo transfers to the fleet model — the fleet layer wraps around it.
